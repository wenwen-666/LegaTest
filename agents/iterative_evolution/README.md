# Iterative Evolution

## Overview

Iterative Evolution is a genetic algorithm–based test suite evolution and optimization system. Through automated crossover, mutation, and selection operations, it continuously optimizes test suites to improve code coverage and test quality.

## 🚀 Key Features

### 🧬 Genetic Algorithm Engine
- Multi-dimensional fitness evaluation: comprehensive scoring based on coverage, diversity, and quality
- Smart evolution strategy: adaptive mutation rate
- Unified management: integrated process management and status monitoring
- Parallel support: optional multi-threaded test execution

### 📊 Coverage Analysis System
- Coverage analysis: line, branch, and method coverage
- JaCoCo report parsing: automatically parse JaCoCo reports generated by Maven
- Surefire reports: integrate Surefire test result analysis
- Fitness calculation: coverage-based composite fitness scoring

### 🎯 Intelligent Optimization System
- Constants configuration: standalone `constants.py` to avoid circular dependencies
- Fast evolution mode: default 10 generations for rapid convergence
- LLM integration: support DeepSeek and other LLMs for smart test generation and repair
- Unified file management: consistent management of test reports and files

## 🚀 Usage Guide

### Prerequisite: CFG extraction with Soot (Step 0)

Before running iterative evolution, extract control-flow graphs (CFG) for the CUT (classes under test) using Soot:

```bash
# Step 0: Extract CFG with Soot (requires Java 11+)
# Output should be placed under: dataset/<project>/soot-output/
# The exact Soot command/script depends on your local setup and project layout.
# Ensure the generated *.cfg.json files are available for the target classes.
```

### Command Line

```bash
# Basic usage - run the full evolution process
python -m agents.iterative_evolution --base_dir /path/to/project --project <project_name>

# Specify number of generations
python -m agents.iterative_evolution \
  --base_dir /path/to/project \
  --project <project_name> \
  --max_gen 10

# Specify start and end generations
python -m agents.iterative_evolution \
  --base_dir /path/to/project \
  --project <project_name> \
  --start_gen 1 \
  --end_gen 5

# Target a single class
python -m agents.iterative_evolution \
  --base_dir /path/to/project \
  --project <project_name> \
  --target_class <target_class> \
  --max_gen 8

# Set fitness threshold and branch target
python -m agents.iterative_evolution \
  --base_dir /path/to/project \
  --project <project_name> \
  --fitness_threshold 0.95 \
  --branch_target 95.0

# Force overwrite existing test reports
python -m agents.iterative_evolution \
  --base_dir /path/to/project \
  --project <project_name> \
  --force_overwrite
```

## 🧮 Core Algorithms

### Fitness Calculation

The system calculates fitness based on coverage, considering the following indicators:

```python
# Fitness calculation (via CoverageAnalyzer)
def calculate_fitness(self, line_coverage, branch_coverage, method_coverage):
    """Compute composite fitness"""
    # Line coverage weight 60%
    line_score = line_coverage * 0.6
    
    # Branch coverage weight 30%
    branch_score = branch_coverage * 0.3
    
    # Method coverage weight 10%
    method_score = method_coverage * 0.1
    
    return line_score + branch_score + method_score
```

### Adaptive Mutation Rate

```python
# Mutation rate calculation (via MutationOperator)
def calculate_mutation_rate(self, generation, fitness):
    """Compute adaptive mutation rate"""
    # Base rate
    base_rate = MUTATION_MIN
    
    # Generation-based decay
    decay_factor = math.exp(-MUTATION_DECAY_RATE * generation)

    # Fitness-based adjustment
    if fitness >= 0.8:
        fitness_factor = 0.5  # reduce rate for high fitness
    elif fitness <= 0.3:
        fitness_factor = 1.5  # increase rate for low fitness
    else:
        fitness_factor = 1.0

    # Final mutation rate
    mutation_rate = base_rate * decay_factor * fitness_factor
    return min(mutation_rate, MUTATION_MAX)
```

### Termination Conditions

The system supports multiple termination conditions:

```python
# Termination check
def should_terminate(self, generation, fitness_history, best_fitness):
    """Check if evolution should terminate"""
    # 1. Reached max generations
    if generation >= MAX_GENERATIONS:
        return True, "Reached maximum generations"
    
    # 2. Reached fitness threshold
    if best_fitness >= FITNESS_THRESHOLD and FITNESS_THRESHOLD > 0:
        return True, "Reached fitness threshold"
    
```

## 🔄 Evolution Flow

### Basic Evolution Flow
```
0. Preprocess (CFG via Soot, Java ≥ 11) → 1. Initialization → 2. Discover test classes → 3. Build initial population → 4. Fitness evaluation
                         ↓
8. Convergence check ← 7. Select best individuals ← 6. Crossover/Mutation ← 5. Selection
```

### Detailed Steps
1. Initialization: set evolution parameters, initialize LLM client and unified manager
2. Test discovery: scan the project to find all test classes
3. Initial population: build from existing tests
4. Fitness evaluation: run tests, compute coverage, evaluate fitness
5. Selection: select the best individuals by fitness
6. Crossover/Mutation: perform crossover and mutation on selected individuals
7. Individual selection: choose the best from newly generated individuals
8. Convergence check: verify if termination conditions are met

## 🔧 Troubleshooting

### Common Issues

1. API key error: check `DEEPSEEK_API_KEY` environment variable
2. Test compile failure: ensure generated test code is syntactically correct
3. Missing coverage report: the system will generate test reports automatically; no pre-run needed
4. Insufficient memory: reduce parallel test count or increase JVM heap
5. Maven command failures: check Maven configuration and project dependencies

### Debug Tips

- Check console logs for evolution progress
- Verify that the `test_reports` directory structure is correct
- Ensure Maven tests run successfully
- Monitor API call frequency to avoid rate limits
- Inspect results in the `evolution_process` directory
